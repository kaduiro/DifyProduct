# AInews_for_LINE + DeepResearch 統合アプリ 改善提案書

## 📋 現状分析

### AInews_for_LINE.yml の特徴
- **構造**: 固定3カテゴリ（基盤モデル、ビジネス、技術・規制）を並列検索
- **検索方式**: 1回の実行で3つの固定クエリを実行
- **問題点**: 
  - カテゴリが固定されており、柔軟性が低い
  - 検索の深掘りができない
  - 8カテゴリ全てをカバーできない可能性

### DeepResearch (1).yml の特徴
- **構造**: イテレーションで動的に検索計画を立案・実行
- **検索方式**: LLMが次に検索すべきトピックを決定し、繰り返し検索
- **強み**: 
  - 動的な検索戦略
  - 深掘り調査が可能
  - 検索済みトピックを追跡して重複を回避

## 🎯 改善提案（推奨度順）

### 【推奨度: ⭐⭐⭐⭐⭐】提案1: ハイブリッド型アーキテクチャ（推奨）

#### 概要
DeepResearchの動的検索ロジックをベースに、AInews_for_LINEの8カテゴリ分類とLINE送信機能を統合

#### 実装方針
1. **開始ノード**: 日付範囲、query_suffixを入力変数として定義
2. **イテレーション内**:
   - LLM（検索計画立案）: 8カテゴリのカバー状況を評価し、複合キーワードを生成
   - Tavily Search: 検索実行
   - Variable Assigner: topics, findingsを更新
3. **イテレーション終了後**:
   - Reasoning Model（gemini-2.0-flash-thinking-exp-01-21）: マスターレポート作成
   - LLM（前半4カテゴリ整形）: 大規模言語モデル、ビジネス、研究、規制
   - LLM（後半4カテゴリ整形）: クリエイティブ、実用化、リスク、未来展望
   - HTTP Request（LINE送信）: 2回に分けて送信

#### 推奨理由
✅ **最も効果的**: DeepResearchの柔軟性とAInewsの8カテゴリ分類を両立  
✅ **網羅性向上**: 動的検索により、固定クエリでは拾えない情報も取得可能  
✅ **効率性**: 複合キーワード戦略で1回の検索で複数カテゴリをカバー  
✅ **拡張性**: カテゴリの追加・変更が容易

#### 改善ポイント
- **プロンプト改善**: 8カテゴリのカバー状況を明確に評価するロジックを追加
- **検索戦略**: 複合キーワード生成の精度向上（例: "AI 規制 政策 リスク 倫理"）
- **終了条件**: 8カテゴリ全てが一定の情報量に達したら終了

---

### 【推奨度: ⭐⭐⭐⭐】提案2: カテゴリ別並列検索 + DeepResearch

#### 概要
8カテゴリを2グループに分け、各グループ内でDeepResearchを並列実行

#### 実装方針
1. **開始ノード**: 日付範囲、query_suffixを定義
2. **並列実行**:
   - **グループ1**: 大規模言語モデル、ビジネス、研究、規制 → DeepResearchイテレーション
   - **グループ2**: クリエイティブ、実用化、リスク、未来展望 → DeepResearchイテレーション
3. **統合**: 2つのグループの結果をマージ
4. **LINE送信**: 2回に分けて送信

#### 推奨理由
✅ **並列処理**: 実行時間の短縮が可能  
✅ **カテゴリ分離**: 各グループで独立した検索戦略を実行  
⚠️ **複雑性**: 並列処理の管理が必要

#### 改善ポイント
- **変数管理**: 各グループのfindingsを別々に管理し、最後に統合
- **リソース管理**: 並列実行によるAPI呼び出し数の増加に注意

---

### 【推奨度: ⭐⭐⭐】提案3: 段階的検索戦略

#### 概要
第1段階で全カテゴリを広く検索、第2段階で不足カテゴリを深掘り

#### 実装方針
1. **第1段階（固定検索）**: 8カテゴリそれぞれで1回ずつ検索
2. **第2段階（DeepResearch）**: 情報が不足しているカテゴリを特定し、DeepResearchで深掘り
3. **統合・整形**: マスターレポート作成 → LINE送信

#### 推奨理由
✅ **確実性**: 全カテゴリを必ずカバー  
⚠️ **効率性**: 固定検索が無駄になる可能性  
⚠️ **柔軟性**: 第2段階の判定ロジックが複雑

---

### 【推奨度: ⭐⭐】提案4: 既存構造の最小変更

#### 概要
AInews_for_LINE.ymlの構造を維持し、検索クエリ生成部分のみLLM化

#### 実装方針
1. 3つのTavily Searchツールの前にLLMノードを追加
2. LLMが8カテゴリを考慮した検索クエリを生成
3. 既存の整形・送信ロジックはそのまま使用

#### 推奨理由
⚠️ **実装容易**: 既存構造をほぼ維持  
❌ **効果限定的**: DeepResearchの深掘り機能が使えない  
❌ **柔軟性低**: 固定3検索の制約が残る

## 🔧 プロンプト改善案

### 検索計画立案LLMのプロンプト改善

#### 現状の問題点
- 8カテゴリのカバー状況を明確に評価していない
- 複合キーワード生成の精度が不十分な可能性

#### 改善案

```yaml
systemプロンプト:
あなたはAIニュースの「取材デスク」です。
今週のAI業界の動向を「広く網羅的」に収集するために、次に検索すべきトピックを決定してください。

## 現在の調査状況
- ターゲット期間: {{#query_suffix#}}
- 既に調査したトピック: {{#topics#}}
- 現在の発見事項: {{#findings#}}

## 必須カバーカテゴリ（全8カテゴリ）
以下のリストのうち、まだ情報が不足しているカテゴリを優先してください。
1. 大規模言語モデル・基盤モデル
2. ビジネス・企業動向
3. 研究・技術革新
4. 規制・政策
5. クリエイティブAI
6. 実用化・導入事例
7. リスク・倫理
8. 未来展望

## カテゴリカバー状況の評価
現在のfindingsを分析し、各カテゴリの情報量を評価してください：
- 十分（3件以上の具体的なニュース）: そのカテゴリは優先度を下げる
- 不足（0-2件）: そのカテゴリを優先的に調査

## 検索戦略
- **【重要】効率重視**: 1回の検索で複数のカテゴリをカバーできるよう、関連性の高い2〜3個のカテゴリを組み合わせた「複合クエリ」を作成してください。
  - 良い例: "AI 規制 政策 リスク 倫理" (4, 7を同時調査)
  - 良い例: "Generative AI Creative Future Outlook" (5, 8を同時調査)
  - 良い例: "LLM foundation model business enterprise" (1, 2を同時調査)
- 特定のモデル（Geminiなど）だけでなく、業界全体の動きを広く探すキーワードを選定してください。
- 全8カテゴリの情報が概ね揃ったと判断した場合（各カテゴリ3件以上）、shouldContinueをfalseにしてください。

## 出力形式
必ず以下のJSON形式で出力してください。Markdownのコードブロックは不要です。
{
  "nextSearchTopic": "カテゴリA カテゴリB 複合キーワード",
  "shouldContinue": true,
  "reasoning": "なぜこのキーワードを選んだか、どのカテゴリをカバーするか"
}
```

### マスターレポート作成LLMのプロンプト改善

#### 改善案

```yaml
systemプロンプト:
あなたはAIニュースの「編集長」です。
取材班が集めた大量の検索結果（findings）に基づき、今週のAIニュースの「マスターレポート」を作成してください。

## 入力データ
{{#findings#}}

## 思考プロセス（Thinking）
1. **重複排除**: 同じニュースが複数回出現している場合は統合してください。
2. **信憑性評価**: 
   - 信頼できる情報源（公式発表、主要メディア）を優先
   - 推測や噂レベルのものは除外するか、「噂レベル」と明記
3. **日付確認**: 指定期間（{{#date_range#}}）外の古いニュースは除外してください。
4. **重要度評価**: 業界への影響度が高いニュースを優先的に記載してください。

## 出力要件
- 以下の8つのカテゴリに分類して記述してください。
  [大規模言語モデル, ビジネス, 研究, 規制, クリエイティブ, 実用化, リスク, 未来]
- 各ニュースには必ず「具体的な数値」「固有名詞」「日付」を含めてください。
- 文体は「だ・である」調で、客観的な事実を中心に記述してください。
- マークダウン形式で出力してください。
- 各カテゴリごとに見出しを付けてください。

## 出力フォーマット例
# マスターレポート

## 大規模言語モデル・基盤モデル
- {具体的なニュース1}（2024年1月15日）
- {具体的なニュース2}（2024年1月16日）
...

## ビジネス・企業動向
...
```

## 🚀 ロジック改善案

### 1. カテゴリカバー状況の可視化

#### 現状
- カテゴリのカバー状況が不明確

#### 改善案
- Variable Assignerで、各カテゴリの情報量をカウントする変数を追加
- LLMに「カテゴリ別情報量」を渡し、より正確な判断を可能に

```yaml
# Codeノードでカテゴリ別カウント
def main(findings: list[str]) -> dict:
    category_counts = {
        "llm": 0,
        "business": 0,
        "research": 0,
        "regulation": 0,
        "creative": 0,
        "implementation": 0,
        "risk": 0,
        "future": 0
    }
    # findingsを分析してカテゴリ別にカウント
    # （簡易的なキーワードマッチングまたはLLMで分類）
    return {"category_counts": category_counts}
```

### 2. 検索の早期終了条件

#### 現状
- shouldContinueの判定がLLM任せで不安定

#### 改善案
- Codeノードで客観的な終了条件をチェック
- 全8カテゴリが一定の情報量に達したら強制終了

```yaml
# IF/ELSEノードの条件
- 全カテゴリが3件以上: 終了
- イテレーション回数が上限（例: 5回）に達した: 終了
- shouldContinueがfalse: 終了
```

### 3. 検索結果の品質向上

#### 現状
- Tavily Searchの設定が固定

#### 改善案
- 検索深度を動的に調整
  - 初期: `advanced`（広範囲）
  - 深掘り時: `advanced` + より具体的なクエリ
- max_resultsをカテゴリの不足度に応じて調整

## ⚠️ 注意事項

1. **API呼び出し数の増加**: DeepResearchにより検索回数が増える可能性
   - Tavily APIのレート制限に注意
   - イテレーション回数の上限を設定

2. **実行時間**: イテレーション実行により処理時間が長くなる
   - タイムアウト設定の確認
   - 非同期処理の検討

3. **コスト**: LLM呼び出し回数の増加
   - gemini-2.0-flash（高速・安価）の使用を推奨
   - Reasoning Modelは必要時のみ使用

4. **変数管理**: conversation variablesとloop variablesの使い分け
   - イテレーション内: loop variables
   - イテレーション間: conversation variables
