# **障害報告書：Dify Docker環境構築におけるデータベース起動不全**

## **1\. 概要**

Windows (WSL2) 環境下でのDify構築において、PostgreSQLコンテナ（db\_postgres）が起動に失敗し、再起動ループ（Restart Loop）に陥っている。これにより、APIサーバーやWorkerなど依存する全てのサービスがデータベースに接続できず、アプリケーション全体が機能しない状態である。

## **2\. 環境情報**

* **OS:** Windows 10/11 (WSL2 backend)  
* **ソフトウェア:** Docker Desktop for Windows  
* **対象アプリケーション:** Dify (GitHub main branch)  
* **リソース設定:**  
  * .wslconfig によりメモリ 16GB を割り当て済み。  
  * wsl free \-h にて、**13GB以上の空きメモリ**があることを確認済み（リソース不足ではない）。

## **3\. エラー詳細**

### **3.1. 主なエラーログ (PostgreSQL)**

db\_postgres コンテナの初期化プロセス（initdb）において、以下のエラーが発生し続けている。

Plaintext

db\_postgres-1 | initdb: error: could not access directory "/var/lib/postgresql/data/pgdata": Out of memory  
db\_postgres-1 | initdb: error: directory "/var/lib/postgresql/data/pgdata" exists but is not empty

### **3.2. 連鎖エラー (API / Worker)**

データベースが起動していないため、他のコンテナで接続エラーが発生。

Plaintext

psycopg2.OperationalError: could not translate host name "db\_postgres" to address: Name or service not known

## **4\. 原因分析**

### **根本原因：WSL2/Docker仮想ディスクの不整合・破損**

* **現象の矛盾:** 13GB以上の空きメモリがあるにもかかわらず、ディレクトリへのアクセス時に Out of memory エラーが発生している。  
* **技術的見解:** これはLinuxカーネル（WSL2）がファイルシステム（ext4）上のI/O操作に失敗した際、エラーコードを誤って「メモリ不足」として報告している典型的なケースである。  
* **結論:** Docker Desktopが使用しているWSL2の仮想ディスクファイル（ext4.vhdx）内部で、データの不整合やファイルシステムの破損が発生している可能性が極めて高い。通常のコマンドライン操作（docker compose down \-v や prune）では、このディスク破損レベルの問題は解消できない。

## **5\. 実施済みの対応（解決せず）**

以下の手順を実施したが、エラーは解消されなかった。

1. **リポジトリの再クローン:** コード自体の問題を疑い実施したが、永続化データ（Volume）が残っていたため改善せず。  
2. **コンテナとボリュームの削除 (down \-v):** コンテナレベルでのデータ削除を試みたが、エラーが継続。  
3. **システム全削除 (system prune \-a):** 未使用データを一掃したが、破損箇所は修復されず。  
4. **WSL2メモリ拡張:** .wslconfig で16GBへ拡張し、WSL再起動を実施。メモリ不足の線は完全に否定された。

## **6\. 推奨される解決策（Next Action）**

**Docker Desktopのデータ領域の物理的消去（Purge）**

コマンドラインからの操作ではなく、Docker DesktopのGUI機能を使用して、WSL2上のデータストアを初期化する必要がある。

### **手順**

1. **Docker Desktop** のGUIを開く。  
2. 右上の **\[Troubleshoot\]** (虫のアイコン) をクリック。  
3. **\[Clean / Purge data\]** を選択。  
4. **"Hyper-V / WSL2 data"** にチェックを入れ、**\[Delete\]** を実行する。

この操作により、破損した仮想ディスクが再生成され、正常にデータベースの初期化が行えるようになる見込みである。
